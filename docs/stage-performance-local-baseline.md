# 本地分阶段性能基线（2026-07-19）

用 `scripts/bench/staged.sh` 跑的**两维分阶段**基线：横向扩展 + 纵向加压。目的是
定位扩展性与 SLO 拐点，**不是**生产容量承诺。

## 环境

- 机器：Apple Silicon 10 核，macOS
- Docker 栈**已完全拆除**（`docker compose down -v`），运行时 `load1 ≈ 3.1`
- 残留负载来自 WindowServer + 运行测试所在的 GUI 应用，约占 2–3 核，**消不掉**；
  因此绝对峰值仍低于完全空闲的 10 核基线（历史记录 6 节点 11.4M/s）
- 撮合无法绑核（Apple Silicon KERN_NOT_SUPPORTED），尾延迟含调度噪声
- workload：`cluster_throughput` / `latency` bench 自带流，成交率约 67%（高交叉，偏重）

## Axis 1 — 横向扩展（cluster_throughput，1M 单/节点，纯内存）

| 节点数 | 聚合 orders/s | 每节点 orders/s | 5M/s |
|------:|-------------:|---------------:|:----:|
| 1 | 2,840,585 | 2,840,585 | ✗ |
| 2 | 4,811,417 | 2,405,709 | ✗ |
| 4 | 6,563,123 | 1,640,781 | ✓ |
| 6 | 8,675,519 | 1,445,920 | ✓ |
| 8 | 9,037,410 | 1,129,676 | ✓ |
| 6（+journal 持久） | 4,893,061 | 815,510 | 接近 |

- 近线性扩展到本机核数上限；4 节点起过 5M/s。
- 每节点吞吐随节点数下降（2.84M→1.13M）是**过订阅签名**：~7 可用核上，
  8 节点已见边际递减，聚合上限约 **9M orders/s**。
- 持久化（journal）走 Mac 本地 APFS fsync，仍非服务器 NVMe，真机会更高。

## Axis 2 — 纵向加压（latency，单分片，1M 单/档）

| 目标/s | 达成/s | p50 | p90 | p99 | 判读 |
|-------:|-------:|----:|----:|----:|------|
| 200k | 199,998 | 34.0µs | 60.7µs | 0.59ms | 从容 |
| 500k | 499,990 | 34.3µs | 61.0µs | 1.68ms | 从容 |
| 1M | 999,910 | 35.5µs | 62.9µs | 4.81ms | 稳 |
| 2M | 1,999,942 | 42.9µs | 1.83ms | 9.92ms | 拐点前 |
| 3M | 2,874,076 | 2.60ms | 5.81ms | 14.93ms | 追不上，排队 |
| 4M | 3,059,943 | 15.24ms | 25.22ms | 29.26ms | 饱和 |

- 单分片守 SLO（p50 ≤ 43µs）到 **~2M/s**；拐点在 2–3M/s；硬饱和 **~3M/s**
  （给到 4M 也只做到 3.06M，延迟塌方）。
- p99 尾部主要是 macOS 调度噪声 + 单分片无法绑核，真机绑核后会紧很多
  （历史记录 @200k/s p99=19µs）。

## 结论

- 引擎**扩展性与确定性完好，无回归**（环境越干净数字越高：6 节点 争抢 7.60M →
  停 docker 7.83M → 全拆 8.68M）。
- **绝对峰值与持久化验收数字本机拿不到**——必须上专用空闲 NVMe 机器跑
  `scripts/acceptance/run-ladder.py`（TPS 阶梯 + 故障注入 + 排空判定）。
- 单一峰值数字会掩盖 SLO 拐点，**分阶段**才能回答"能稳定支撑多少 TPS"。

> 复现：`scripts/bench/staged.sh`（可用 `NODES` / `RATES` / `ORDERS_PER_*` /
> `JOURNAL` 环境变量调参）。运行前先停掉 docker 栈并确认 `load1` 低。
